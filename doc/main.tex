\section{Propuesta}

Aprender más sobre Infrastructure as Code (IaC) usando herramientas como OpenTofu que permite describir y desplegar infraestructura (máquinas virtuales, redes, almacenamiento, etc.) de forma declarativa, mientras que NixOS (tenemos experiencia previa) proporciona un sistema operativo completamente reproducible cuya configuración también se define mediante código.

Este trabajo propone explorar la integración de ambas tecnologías para crear una infraestructura completamente declarativa, donde tanto la creación de recursos como la configuración del sistema operativo estén definidas en repositorios versionados (git).
Objetivos

El objetivo principal del trabajo es diseñar e implementar una plataforma de infraestructura reproducible basada en principios de Infrastructure as Code.

\section{Objetivos}

\begin{itemize} 
  \item Estudiar el paradigma de infraestructura declarativa.
  \item Analizar el funcionamiento de OpenTofu para la provisión de recursos.
  \item Analizar el modelo de configuración declarativa de NixOS.
  \item Diseñar una arquitectura donde OpenTofu gestione la infraestructura y NixOS configure los sistemas.
  \item Implementar un entorno reproducible donde la infraestructura pueda desplegarse automáticamente a partir del código.
  \item Evaluar las ventajas y limitaciones del enfoque propuesto.
\end{itemize} 

\section{Tecnologías usadas}

\begin{itemize}
  \item Proxmox VE: plataforma de virtualización open-source que utilizamos como hipervisor para la gestión de nuestras máquinas virtuales.
  \item OpenTofu: fork de Terraform open-source, herramienta de IaC que empleamos para la delcaración y asiagnación de recursos de nuestras máquinas virtuales.
  \item NixOS: distribución de Linux que emplea el gestor de paquetes Nix, gracias a su configuración delcarativa nos permite desplegar en las máquinas virtuales todo tipo de servicios de forma sencilla y automatizada.
  \item nixos-anywhere: herramienta de instalación de NixOs, nos permite realizar la instalación del SO de forma remota mediante ssh y de forma completamente automatizada.
  \item Colmena: tras la primer build de los sistemas de NixOS Colmena nos permite realizar cambios sobre el SO (mientras este se ejecuta) de forma remota y sencilla sin falta de reinstalar.
  \item agenix: paquete que permite la gestion de secretos mediante encriptación age (\url{https://github.com/FiloSottile/age}), de esta forma podemos mantener registro de claves y compartirlas entre máquinas sin miedo a exponernos.
  \item Docker: mediante la configuración en nixos, podemos desplegar servicios sobre docker en boot lanzandolos de forma automatizada.
\end{itemize}

\section{Diagrama de red}

Para comenzar el proyecto decidimos que lo primero de todo era deifinir la estructura de red, y equipos/contenedores a crear.

Como se puede ver en la \hyperref[fig:red]{Figura \ref{fig:red}}, se creo una red DMZ que conectase con el router para poder aislar 
las máquinas y reproducir de forma más fiable una red empresarial. La red LAN es la red Wi-Fi del router y la WAN representa internet.
El router realmente está virtualizado al igual que el resto de contenedores y VM's. Hemos redireccionado el puerto 3478 del router para que 
permita la conexión con la vpn que montamos. La redireccion del puerto 80/443 hacia la máquina con caddy permite el acceso externo a la infraestructura.

El resto de las máquinas de la red estan aisladas frente accesos no permitidos, śolamente acceptan conexiones ssh con certificado (se introducieron dentro de las VM
nuestros certificados para un acceso más seguro y rápido).

Una vez con esto establecido podíamos centrarnos en el desarrollo del IaC.
